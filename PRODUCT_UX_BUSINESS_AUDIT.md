# Product / UX / Business Audit

Дата: 2026-05-09  
Проект: Matchday Store, интернет-магазин атрибутики ФК «Шинник»  
Фокус: бизнес-логика, UX, операционная работа, администрирование и ощущение production-продукта.

Аудит основан на статическом просмотре кода, шаблонов, моделей, dashboard, checkout, email-flow, складской логики и deployment-конфигурации. Тесты и локальный сервер в рамках этого аудита не запускались.

## 1. Сильные стороны проекта

- Складская логика для MVP сделана сильнее среднего: есть физический остаток, резерв, доступный остаток, атомарный резерв и списание только при фактической выдаче заказа.
- Checkout защищен от ряда реальных проблем: есть idempotency token, блокировки пользователя/корзины/SKU, лимит активных неоплаченных заказов, лимит количества одного SKU, проверка подтвержденного email.
- Есть staff-dashboard, а не только Django admin. Менеджер может видеть заказы, менять статусы, отмечать оплату и управлять складом.
- Статусы заказа не просто строки в модели: есть отдельный staff-flow, допустимые переходы, запрет выдачи без оплаты, запрет изменения финальных заказов.
- Есть журнал переходов статусов заказа: кто, когда и что поменял. Это важно для операционного контроля.
- Есть email-уведомления клиентам и сотрудникам: заказ создан, отменен, готов, оплата подтверждена, новый заказ для staff.
- Есть автоотмена просроченных заказов самовывоза и снятие резерва через доменный сервис.
- Deployment выглядит production-like: Docker, nginx, gunicorn, PostgreSQL, Redis, Celery worker/beat, healthchecks, log rotation, runbook, backup scripts, Sentry/env/security settings.
- Есть smoke E2E на полный sales-flow: регистрация, checkout, уведомления, dashboard, выдача.

## 2. Главные UX / бизнес-проблемы

### P0. Checkout требует аккаунт и подтвержденный email до покупки

Для модели "самовывоз + оплата при получении" это высокий friction. Покупатель может быть готов оставить телефон и забрать товар, но система требует регистрацию, вход и подтверждение email. Если письмо не пришло, попало в spam или пользователь не хочет создавать аккаунт, продажа теряется.

Риск: падение конверсии на самом дорогом этапе воронки.

Что сделать:

- Рассмотреть гостевой checkout.
- Либо сделать "быстрый заказ" с телефоном и email без предварительного подтверждения, а аккаунт создавать после заказа.
- Если подтверждение email остается обязательным, интерфейс должен заранее объяснять это до корзины/checkout.

### P0. В checkout есть технический текст, видимый покупателю

В блоке "Важно" показаны формулировки уровня продуктовой заметки: "После оформления пользователь должен ясно видеть итог покупки", "Checkout должен быть простым и не вызывать сомнений". Это не пользовательский текст, а внутренний UX checklist.

Риск: проект сразу выглядит как pet-project или учебная работа.

Что сделать:

- Заменить блок на коммерческую информацию:
  - "Оплата при получении".
  - "Самовывоз из магазина ФК «Шинник»".
  - "Забрать заказ можно после уведомления о готовности".
  - "Резерв хранится N рабочих дней".
  - "Возьмите с собой номер заказа".

### P0. Контакты и условия самовывоза противоречат друг другу

В settings часы самовывоза: пн-пт 10:00-16:00, сб-вс выходной. В футере: ежедневно 10:00-19:00. Email поддержки generic: `support@matchday-store.com`, хотя магазин заявлен как официальный магазин ФК «Шинник».

Риск: покупатели будут приезжать не в то время, сотрудники получат лишние звонки, доверие к магазину падает.

Что сделать:

- Завести единый источник контактов и часов работы.
- Использовать реальные данные клуба/магазина.
- Проверить, что checkout success, email, footer и личный кабинет используют одинаковые данные.

### P0. Складские резервы реализованы технически, но плохо видны менеджеру

В backend есть `quantity`, `reserved_quantity`, `available_quantity`. Но в интерфейсе склада менеджер в основном видит "остаток", без разложения "физически / зарезервировано / доступно".

Риск: менеджер не понимает, почему нельзя уменьшить остаток, где зависли товары, какие заказы блокируют склад.

Что сделать:

- В списке склада и карточке товара показывать:
  - физический остаток;
  - резерв;
  - доступно к продаже;
  - список активных заказов, которые держат резерв по варианту.

### P1. Каталог слишком бедный для реального магазина атрибутики

Сейчас есть поиск, категория и сортировка. Для футбольной атрибутики этого мало.

Не хватает:

- фильтра по размеру;
- фильтра "в наличии";
- фильтра по цене;
- фильтра по типу товара;
- фильтра по полу/возрасту, если появится одежда для детей/женщин;
- меток "новинка", "хит", "последние размеры", "sale";
- нормальной SEO/коммерческой структуры категорий.

Риск: при росте каталога пользователь не сможет быстро найти нужный товар.

### P1. Карточка товара не закрывает покупательские сомнения

Есть фото, название, цена, выбор варианта, описание. Но для реального магазина не хватает:

- размерной сетки;
- состава материала;
- ухода;
- артикула/SKU;
- точного наличия по вариантам;
- фото спереди/сзади/детали/на модели;
- условий получения и оплаты прямо на карточке;
- гарантии оригинальности;
- условий возврата.

Риск: пользователь не уверен в размере, качестве и условиях покупки, поэтому откладывает заказ.

### P1. Dashboard заказов подходит для малого потока, но слаб для ежедневной работы

Сейчас есть фильтры по status и поиск по номеру/email. Для менеджера этого мало.

Не хватает:

- поиска по телефону;
- фильтра по оплате;
- фильтра по дате;
- фильтра по сумме;
- отдельной очереди "требует внимания";
- просроченных/почти просроченных самовывозов;
- bulk-actions;
- печатного листа сборки;
- печати/экспорта заказа;
- быстрых действий без захода в карточку.

Риск: менеджер будет тратить много кликов и пропускать проблемные заказы.

### P1. В карточке заказа менеджера не хватает операционного контекста

Состав, статусы, получатель и телефон есть. Но с точки зрения реальной выдачи не хватает:

- комментария клиента;
- email отдельной строкой;
- кликабельного телефона;
- pickup deadline;
- внутренней заметки сотрудника;
- причины отмены;
- истории коммуникации;
- кнопки "позвонить/написать";
- печати сборочного листа.

Риск: сотрудник будет открывать другие страницы или искать информацию вручную.

### P1. Нет бизнес-аналитики для владельца

Владелец не видит:

- выручку;
- количество заказов;
- средний чек;
- популярные товары;
- продажи по дням;
- отмены;
- зависшие резервы;
- low stock;
- брошенные корзины;
- конверсию;
- динамику продаж.

Риск: магазин работает "вслепую"; невозможно понять, что продавать, что докупать и где теряются клиенты.

### P1. Аудит изменений неполный

Переходы статусов заказа логируются. Но для бизнеса так же важны изменения:

- цены;
- остатков;
- публикации/снятия товара;
- фото;
- описания;
- категорий;
- ручной корректировки заказа.

Риск: невозможно расследовать, кто случайно поменял цену, снял товар или изменил склад.

### P2. Юридический и доверительный контент слишком общий

Юридические страницы есть, но тексты generic. Нет явных реквизитов продавца, конкретного порядка возврата, ответственного лица, фактических условий покупки.

Риск: для настоящего магазина это выглядит неполно и снижает доверие.

### P2. Demo-каталог конфликтует с позиционированием магазина

В fixture есть товары Barcelona, Real Madrid, Manchester United, Juventus, Arsenal, Liverpool. Для магазина ФК «Шинник» это сбивает позиционирование.

Риск: если fixture попадет в demo/prod, магазин выглядит неофициальным и учебным.

Что сделать:

- Заменить fixture на товары ФК «Шинник».
- Либо явно пометить fixture как dev/demo и не использовать в production.

### P2. Заявлен DRF, но фактический API-слой не найден

В зависимостях нет `djangorestframework`, по коду не найдено `APIView`, `ViewSet`, `serializers`. Фактически проект server-rendered Django.

Риск: расхождение между заявленным стеком и реальностью. Это не проблема для MVP, если API не нужен, но в документации лучше не заявлять DRF без фактической реализации.

## 3. Что обязательно добавить до production

### Critical

- Убрать технический текст из checkout.
- Унифицировать контакты, часы работы, телефон, email, pickup point во всех местах.
- Проверить `auth_basic` в nginx: если это production public storefront, global basic auth нужно убрать или ограничить staging/dashboard.
- Показать в checkout и success:
  - оплату при получении;
  - адрес самовывоза;
  - часы работы;
  - срок хранения резерва;
  - что заказ нужно забирать после уведомления о готовности.
- Добавить в dashboard order detail:
  - комментарий клиента;
  - email;
  - кликабельный телефон;
  - pickup deadline;
  - внутренние заметки.
- Добавить в склад:
  - физический остаток;
  - резерв;
  - доступно;
  - активные заказы, удерживающие резерв.
- Убрать/заменить demo-каталог с чужими клубами.

### Important

- Фильтры каталога по размеру, наличию, цене.
- Размерная сетка и атрибуты товара.
- SKU/артикул как реальное поле, а не `SKU-{pk}`.
- Бизнес-дашборд владельца.
- Stock movement ledger.
- История изменений товара/цены/остатка.
- Bulk-actions для заказов.
- Печатный лист сборки.
- Причина отмены заказа.
- Напоминание о готовом заказе перед автоотменой.

### Nice-to-have

- Гостевой checkout.
- Промокоды и акции.
- Подборки товаров.
- Wishlist / избранное.
- Отзывы или хотя бы trust-блоки.
- Онлайн-оплата.
- Доставка/PVZ.
- DRF/API, если это действительно часть продуктовой цели.

## 4. Что больше всего улучшит ощущение настоящего магазина

1. Реальный контент ФК «Шинник»: товары, фото, контакты, email, юридические данные.
2. Коммерческая карточка товара: размер, состав, уход, наличие, фото, условия возврата.
3. Честный checkout: самовывоз, оплата при получении, срок хранения резерва, понятные следующие шаги.
4. Staff-dashboard как рабочий инструмент, а не CRUD:
   - очередь заказов;
   - проблемные заказы;
   - быстрые статусы;
   - печать сборки;
   - внутренние заметки;
   - фильтры.
5. Минимальная аналитика владельца:
   - заказы сегодня;
   - выручка сегодня/неделя/месяц;
   - топ товаров;
   - low stock;
   - отмены;
   - зависшие резервы.

## 5. Что будет раздражать

### Покупателей

- Обязательная регистрация и подтверждение email перед покупкой.
- Непонятно, когда товар реально зарезервирован.
- Нет размерной сетки.
- Нет точного наличия по размерам.
- Нет ясного срока хранения заказа.
- Разные часы работы в разных местах.
- Generic email и тексты `Matchday Store` вместо сильного клубного бренда.
- Нет явной информации о возврате прямо в покупательском пути.

### Менеджеров

- Слишком мало фильтров по заказам.
- Нельзя быстро найти заказ по телефону.
- Нет bulk-actions.
- Нет печати сборочного листа.
- Комментарий клиента не выделен в dashboard.
- Нет внутренней заметки.
- Не видно, какие заказы держат резервы.
- Раздельные формы "изменить статус" и "изменить оплату" требуют лишних кликов.

### Администратора / владельца

- Нет бизнес-аналитики.
- Нет отчета по продажам.
- Нет отчета по отменам.
- Нет истории изменения цены/остатка/товара.
- Нет контроля эффективности сотрудников.
- Нет управления акциями.
- Нет видимости брошенных корзин.

## 6. Сырые бизнес-процессы

### Обработка заказа

Backend-flow сильный, но операционный UX еще сырой. Нужны очереди, быстрые действия, проблемные статусы, просрочка самовывоза, печать сборки.

### Склад

Техническая модель хорошая, но интерфейс не раскрывает резервы и доступность. Для сотрудника это критично.

### Возвраты

Статусы `REFUNDED` / `RETURNED` есть, но полноценного процесса возврата нет: причина, комментарий, сумма, кто подтвердил, что вернулось на склад.

### Поддержка клиентов

Нет истории коммуникации, внутренних заметок, причины отмены, шаблонов сообщений.

### Контент магазина

Контентные страницы есть, но они generic. Для production нужны конкретные условия продавца.

### Маркетинг

Нет акций, промокодов, подборок, abandoned cart, рекомендаций, sale-полей в модели.

## 7. Где проект уже сильнее типичного junior pet-project

- Есть полноценная модель заказа с раздельными статусами заказа, оплаты и исполнения.
- Есть отдельная модель платежей.
- Есть складской резерв, а не простое списание при checkout.
- Есть защита от oversell через транзакции и `F()`-выражения.
- Есть idempotency checkout.
- Есть автоотмена просроченных заказов.
- Есть staff-dashboard.
- Есть журнал переходов статусов.
- Есть Celery email-flow и fallback.
- Есть production runbook и healthchecks.
- Есть rate limits.
- Есть CSP/security settings.
- Есть backup/restore scripts.

## 9. Приоритетный порядок внедрения

1. Сначала закрыть явные production-feeling проблемы: checkout text, контакты, demo-каталог, условия самовывоза.
2. Затем улучшить staff-dashboard: комментарии, телефон, фильтры, резервы, deadline.
3. После этого расширить карточку товара и фильтры каталога.
4. Далее добавить owner analytics и audit logs.
5. Только после стабилизации MVP брать онлайн-оплату, доставку, промокоды и API.

## 10. Главный вывод

Проект уже не выглядит как простой junior CRUD: складской резерв, статусы, транзакции, уведомления, dashboard и deployment сделаны на хорошем уровне для MVP.

Главная проблема сейчас не в backend-ядре, а в продуктовой полноте и операционном UX. Система умеет принять и обработать заказ, но покупателю пока не хватает доверия и ясности, менеджеру - рабочих инструментов для потока заказов, владельцу - аналитики и контроля бизнеса.

Если закрыть checkout/friction, привести контент к реальному клубному магазину, показать резервы менеджеру и добавить минимальную аналитику, проект начнет ощущаться как настоящий production-магазин, а не учебный e-commerce.
